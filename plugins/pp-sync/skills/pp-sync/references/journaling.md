# Automated Journaling & Project Board Integration

The `pp` CLI includes a built-in "Work Journal" system that tracks tasks, technical reasoning, and audit findings, automatically documenting your work for the dev team and project managers.

## Workflow

1.  **Init**: One-time setup for a project.
    ```bash
    pp journal <project> init
    ```
    This creates a `JOURNAL.md` in your repository root.

2.  **Open**: Start a new task.
    ```bash
    pp journal <project> open "Implement Applicant Dashboard"
    ```
    If configured, this will automatically create an Issue on GitHub (`gh`) or GitLab (`glab`) and record the starting point in the journal.

3.  **Note**: Document reasoning as you work.
    ```bash
    pp journal <project> note "Fixed FetchXML N+1 query by pulling the lookup outside the loop."
    ```
    This appends a structured entry to `JOURNAL.md` and posts it as a comment on the GitHub/GitLab issue.

4.  **Close**: Finalize the task.
    ```bash
    pp journal <project> close "Dashboard implemented with optimized FetchXML."
    ```
    This posts a final summary, links the work, and automatically closes the GitHub/GitLab issue.

## Configuration

During `pp setup` or `pp project add`, you will be prompted for:
- **Project Board URL/ID**: The mapping for the project.
- **System**: `github`, `gitlab`, or `auto`. `auto` infers the system from the board URL when possible.
- **AI Attribution**: Whether to explicitly tag notes as `[AI-Assisted]`.

## Safety & Attribution

- **Session Verification**: On first use of each distinct board mapping in a session, `pp` will verify the board mapping to prevent cross-posting.
- **Same-repo URL enforcement** *(v2.7.0+)*: Issue URLs harvested from `JOURNAL.md` are validated against the current repo before any `gh issue comment` / `gh issue close` / `glab` call. Cross-repo URLs (e.g. an attacker-inserted link to a different repo's issue via a malicious PR) are rejected with a clear error before any state-changing action runs. The check uses a strict URL-shape regex plus a host+path-prefix match against `gh repo view`. Tested by `plugins/pp-sync/tests/test_journal_url_validation.sh` (16 cases including subdomain spoofs, port injection, http downgrade, and prefix-confusion attacks).
- **AI-Assisted Tagging**: When enabled, the tool will automatically append ` [AI-Assisted]` to every note and comment, ensuring your team knows what was generated by the agent.
- **Traceability**: Every entry in `JOURNAL.md` and every board comment provides a permanent record of the *why* behind your technical decisions.

## Platform note

The journaling workflow is part of the Bash-based `pp` CLI, so it currently follows the same macOS/Linux/WSL support boundary as the rest of `pp-sync`. Native Windows support is planned as a separate release path.
